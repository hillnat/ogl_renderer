Convenience function to export directly to a file. Use \#\+Set\+IOSystem to supply a custom \mbox{\hyperlink{class_assimp_1_1_i_o_system}{IOSystem}} to gain fine-\/grained control about the output data flow of the export process. 
\begin{DoxyParams}{Parameters}
{\em p\+Blob} & A data blob obtained from a previous call to \mbox{\hyperlink{include_2assimp_2cexport_8h_a9615510b8430a9da4f435a72148128dd}{ai\+Export\+Scene}}. Must not be nullptr. \\
\hline
{\em p\+Path} & Full target file name. Target must be accessible. \\
\hline
{\em p\+Preprocessing} & Accepts any choice of the \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410}{ai\+Post\+Process\+Steps}} enumerated flags, but in reality only a subset of them makes sense here. Specifying \textquotesingle{}preprocessing\textquotesingle{} flags is useful if the input scene does not conform to \mbox{\hyperlink{namespace_assimp}{Assimp}}\textquotesingle{}s default conventions as specified in the \mbox{\hyperlink{}{Data Structures Page }}. In short, this means the geometry data should use a right-\/handed coordinate systems, face winding should be counter-\/clockwise and the UV coordinate origin is assumed to be in the upper left. The \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410a133fd1162674e68bf8cd17070898a936}{ai\+Process\+\_\+\+Make\+Left\+Handed}}, \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410a06922b6a1f1cd8186f9fdafb471c813e}{ai\+Process\+\_\+\+Flip\+UVs}} and \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410a429a11bf7ace46f039f55de895505d4a}{ai\+Process\+\_\+\+Flip\+Winding\+Order}} flags are used in the import side to allow users to have those defaults automatically adapted to their conventions. Specifying those flags for exporting has the opposite effect, respectively. Some other of the \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410}{ai\+Post\+Process\+Steps}} enumerated values may be useful as well, but you\textquotesingle{}ll need to try out what their effect on the exported file is. Many formats impose their own restrictions on the structure of the geometry stored therein, so some preprocessing may have little or no effect at all, or may be redundant as exporters would apply them anyhow. A good example is triangulation -\/ whilst you can enforce it by specifying the \mbox{\hyperlink{include_2assimp_2postprocess_8h_a64795260b95f5a4b3f3dc1be4f52e410a9c3de834f0307f31fa2b1b6d05dd592b}{ai\+Process\+\_\+\+Triangulate}} flag, most export formats support only triangulate data so they would run the step even if it wasn\textquotesingle{}t requested.\\

Supplies a custom IO handler to the exporter to use to open and access files.

If you need \mbox{\hyperlink{class_assimp_1_1_exporter_a03adf017791ffc3694e382678f056e9c}{Export}} to use custom IO logic to access the files, you need to supply a custom implementation of \mbox{\hyperlink{class_assimp_1_1_i_o_system}{IOSystem}} and IOFile to the exporter.

\mbox{\hyperlink{class_assimp_1_1_exporter_ac45a55fc178256576d2d21b58bd944a0}{Exporter}} takes ownership of the object and will destroy it afterwards. The previously assigned handler will be deleted. Pass NULL to take again ownership of your \mbox{\hyperlink{class_assimp_1_1_i_o_system}{IOSystem}} and reset \mbox{\hyperlink{namespace_assimp}{Assimp}} to use its default implementation, which uses plain file IO.
